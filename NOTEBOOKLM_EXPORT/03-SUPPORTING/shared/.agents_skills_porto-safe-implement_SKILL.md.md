# Objetivo

Executar mudanças no projeto Porto com escopo controlado, minimizando improviso, regressão e leitura desnecessária da codebase.

# Quando esta skill deve ser usada

Use esta skill quando o pedido envolver um ou mais destes cenários:

- implementar ou ajustar feature existente
- alterar frontend ou backend já existentes
- revisar fluxo antes de editar
- melhorar robustez de parser, endpoint, componente ou tela
- validar implementação com evidência
- mapear arquivos centrais antes de modificar

Não use esta skill para:

- brainstorming genérico
- refatoração arquitetural ampla sem pedido explícito
- criação de projeto do zero
- pesquisa ampla em todo o repositório sem escopo

# Regras obrigatórias

1. Trabalhe com base no código real do projeto.
2. Não invente endpoints, componentes ou fluxos paralelos sem necessidade clara.
3. Não use mocks se o pedido for aderência ao sistema atual.
4. Não use `as any` nem tipagem improvisada.
5. Não trate regra de negócio existente como bug sem evidência explícita.
6. Preserve comportamento atual fora do escopo solicitado.
7. Não faça buscas amplas e barulhentas em arquivos gerados.

# Escopo de leitura padrão

Priorize apenas estas áreas, salvo evidência de dependência direta fora delas:

- `frontend/src/app`
- `frontend/src/components`
- `frontend/src/context`
- `frontend/src/lib`
- `frontend/src/types`
- `api-server/src`

Evite pesquisar ou depender de:

- `node_modules`
- `.next`
- `dist`
- `coverage`
- `build`
- `*.tsbuildinfo`

# Fluxo obrigatório de trabalho

## Etapa 1 — Entendimento

Antes de editar qualquer arquivo:

- identifique os arquivos centrais da tarefa
- diferencie arquivo central de arquivo secundário
- resuma o fluxo real antes de propor mudança

Formato obrigatório desta etapa:

- arquivo
- componente/função
- responsabilidade
- impacto visual
- impacto funcional

## Etapa 2 — Plano de alteração

Antes de editar:

- liste exatamente quais arquivos pretende alterar
- explique em uma linha por que cada arquivo precisa mudar
- diga explicitamente o que NÃO será alterado

## Etapa 3 — Implementação

Ao implementar:

- faça a menor mudança capaz de resolver o problema
- reaproveite parser, endpoint, componente e tipos já existentes
- mantenha consistência com padrões do projeto
- não crie solução paralela se o projeto já tiver utilitário equivalente

## Etapa 4 — Validação

Depois de editar:

- rode as validações disponíveis no projeto
- liste os comandos executados
- se não houver testes suficientes, diga isso claramente
- diferencie:
  - validado por execução
  - validado por inspeção
  - não validado

## Etapa 5 — Entrega

A resposta final deve conter:

1. arquivos alterados
2. resumo objetivo do que mudou
3. comandos executados
4. evidências de validação
5. riscos restantes
6. veredito final:
   - PRONTO PARA TESTE
   - PRONTO COM RESSALVAS
   - NÃO PRONTO

# Regras para respostas

Nunca diga apenas:

- "está certo"
- "deve funcionar"
- "aparentemente está ok"

Sempre dê evidência.

Se não executou algo, diga explicitamente:

- "validado por inspeção"
- "não foi executado"
